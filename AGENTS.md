# Prototype Instructions

Run the local server yourself and open the preview in the browser available to this environment. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Durable decisions

- Framework: Vue 3 single-file components with Vite.
- Styling: Tailwind CSS v4 through the official Vite plugin, with shared tokens in `src/styles.css`.
- Visual source of truth: `design-reference.png` (selected ideation direction 2).
- Product structure: keep this prototype self-contained inside `landing/` so future product areas can live in sibling folders.
- Social app: keep the signed-in Vue experience in `app/`, built under `/app/` and visually derived from the landing tokens.
- API: keep same-origin Worker endpoints in `api/`; use D1 through the logical `DB` binding for durable profiles, recipes, likes, bookmarks, comments, and follows.
- Authentication: recetitas.app owns registration and login with email/handle plus password. Passwords use salted PBKDF2 hashes; opaque sessions are stored hashed in D1 and sent only through HttpOnly same-site cookies.
- Content integrity: production and local previews start without synthetic recipes; every visible community recipe must be created by a signed-in user.
- Profile navigation: public profiles use stable `/app/u/:handle` URLs; profile edits, follow relationships, connection lists, and owned-recipe deletion remain same-origin API operations.
- Recipe media: browsers upload only through the authenticated same-origin API; the Worker forwards validated JPEG/PNG/WebP bytes to the VPS media service, while D1 stores only the resulting HTTPS URL.
- Recipe video: recipes may contain a cover photo, a video, or both. Videos may be selected or recorded in-browser, are limited to 35 seconds and 50 MB before processing, and the VPS validates duration then transcodes them to MP4 H.264. The feed prefers the photo cover; recipe detail leads with the video and keeps the result photo below. Playback includes seek, fullscreen, 1×/1.5×/2× speed, and press-and-hold temporary 2×.
- Discovery: hashtags are persisted in the `recipe_tags` D1 table and drive the real Discover view, including tag filters, trending counts, community creators, follow actions, recipe search, and active live streams ordered by viewers and recency. Live previews connect lazily only while visible and open the broadcaster profile for full playback and interaction. Keep Discover out of the mobile bottom bar; expose it from the mobile header while the bottom bar remains Inicio, Guardadas, Perfil.
- Recipe edits: only the author may edit a recipe. Every edit requires a short public note, increments the recipe's edit count, updates `updated_at`, and appends to the durable `recipe_edits` history.
- Notifications: persist recipient-scoped like, comment, and follow notifications in D1; never notify users about their own actions. Show an unread badge, refresh it only on session/focus/visibility, and load the full panel on demand.
- App performance: keep secondary views and panels async-loaded, avoid video metadata preloads in feed cards, and preserve offscreen recipe-card rendering containment. The live Core Web Vitals audit remains unavailable until `chrome-devtools` MCP is configured.
- Live profiles: profile owners can broadcast real-time WebRTC video through the VPS MediaMTX service using WHIP; viewers consume WHEP directly in the profile. D1 owns live metadata, broadcaster/viewer heartbeats, likes, comments, moderators, and chat bans. A stream remains `starting` and invisible to viewers until the WHIP publisher connects and explicitly marks it ready; publisher loss ends the live record to prevent ghost WHEP sessions. Owners may delete messages, grant/revoke moderators, and ban/unban chat users; moderators may delete and ban but cannot manage the owner or other moderators. Camera controls must request browser permission, allow video-only fallback when the microphone is unavailable, expose actionable validation instead of silently disabled start controls, and release the active video track before front/rear switching. On multi-lens phones the UI exposes only Frontal/Trasera and prioritizes the lowest-numbered main lens in each group so the rear camera with torch support is selected before auxiliary lenses. Microphone/video toggles and torch remain available only when supported. Publish tokens are stored hashed and validated server-to-server through the media service.
- Deployment caching: application HTML is always served with `Cache-Control: no-cache, no-store`; missing asset URLs return a real 404 instead of the SPA HTML fallback, and Vite preload failures trigger one cache-busting reload so stale dynamic chunks recover after a deployment.
- Avatar system: keep the generated 8×8 pig-chef atlas as the source, slice it with `scripts/slice-avatar-atlas.py`, and serve the 64 individual WebP files from `app/public/avatars`; every user receives a stable index from 0–63 and may reroll it from profile editing.
- Assisted cooking: recipe detail owns an automatic ingredient checklist, a full-screen step-by-step mode, an accurate pauseable timer, and device-local temporary cooking progress. Ingredient discovery ranks only real community recipes against natural-language pantry input.
- Recipe metadata: every recipe declares an easy/medium/hard difficulty and Spanish/English/Portuguese content language; language filters never auto-translate user-authored recipe text.
- Social recipe tools: optional recipe polls, public profile collections, comment photos, deterministic recipe-of-the-day selection, and derived public achievements are durable D1-backed product data. Comment image bytes continue through the authenticated VPS media service while D1 stores only the URL.
- Accessibility: interface language, text scale, color-vision palettes, high contrast, and reduced-motion preferences are device-local settings applied at the document root.
